# AES-256-GCM 加解密原理

本工具基于浏览器原生 Web Crypto API，使用 AES-256-GCM 对称加密。所有运算在浏览器本地完成，密码和明文不经过任何服务器。

下面拆解整个加解密链路。

---

## 整体流程

```
加密：明文 + 密码 → PBKDF2 密钥派生 → AES-GCM 加密 → Base64(salt + iv + ciphertext)
解密：Base64 密文 + 密码 → 拆出 salt/iv/ciphertext → PBKDF2 还原密钥 → AES-GCM 解密 → 明文
```

---

## 1. 密钥派生：PBKDF2

用户输入的密码是变长字符串，不能直接用作 AES 密钥。需要通过 PBKDF2（Password-Based Key Derivation Function 2）将密码派生为固定 256 位的密钥。

```
PBKDF2(password, salt, iterations=100000, hash=SHA-256) → 256-bit AES Key
```

两个关键参数：

**Salt（16 字节随机值）**

每次加密时随机生成。作用是确保相同密码在不同加密场景下派生出不同密钥。没有 salt 的话，攻击者可以预计算一张「密码 → 密钥」的彩虹表，拿到密文后直接查表。加了随机 salt，彩虹表失效——攻击者必须针对每个 salt 单独计算。

**迭代次数（100,000 次）**

PBKDF2 内部会将密码和 salt 反复做 HMAC-SHA256 运算 10 万次。这是一个刻意引入的计算成本：对正常用户来说只慢几十毫秒，但对暴力破解者来说，每次猜测都要付出同样的代价，大幅提高穷举攻击的时间成本。

## 2. AES-256-GCM 加密

拿到 256 位密钥后，进入加密环节。

**AES（Advanced Encryption Standard）** 是分组密码，以 128 位为一个块进行加密。256 表示密钥长度为 256 位。

**GCM（Galois/Counter Mode）** 是 AES 的一种工作模式，本质上是 CTR 模式（计数器模式）+ GMAC（认证码）的组合：

- **CTR 部分**：将递增的计数器与密钥加密后生成密钥流，再与明文 XOR 得到密文。这使得 AES 可以像流密码一样工作，不需要填充（padding）。
- **GMAC 部分**：在加密的同时，基于 Galois 域乘法计算出一个 128 位的认证标签（Authentication Tag）。这个标签附在密文末尾。

GCM 相比 CBC 等模式的优势：
- **认证加密（AEAD）**：一步完成加密 + 完整性校验，不需要额外的 HMAC
- **密文篡改检测**：解密时会校验认证标签，任何 1 bit 的改动都会导致解密失败，而不是输出错误的明文
- **可并行化**：CTR 模式天然支持并行加密，性能好

**IV（12 字节随机值）**

初始化向量，每次加密随机生成。IV 的作用是保证即使相同的密钥加密相同的明文，产出的密文也不同。GCM 模式下推荐 IV 长度为 12 字节（96 位）。

> 需要注意：在同一密钥下，IV **绝对不能重复**。IV 重复（nonce reuse）会导致密钥流重复，攻击者可以通过 XOR 两段密文恢复明文。本工具每次加密都随机生成 salt，因此每次派生的密钥不同，IV 碰撞的风险可以忽略。

## 3. 输出格式

加密完成后，salt、IV、密文（含认证标签）拼接为一个字节数组，再 Base64 编码输出：

```
┌────────────┬──────────┬──────────────────────────────┐
│ salt       │ iv       │ ciphertext + auth tag        │
│ 16 bytes   │ 12 bytes │ 原文长度 + 16 bytes(tag)     │
└────────────┴──────────┴──────────────────────────────┘
                    ↓ Base64 编码
          一串 ASCII 字符串，可安全复制传输
```

salt 和 IV 是公开信息，不需要保密。它们的安全价值在于**随机性**而非**保密性**。密文的安全性完全依赖于密码（密钥）的保密。

## 4. 解密过程

解密是加密的逆操作：

1. Base64 解码，按固定偏移量拆出 salt（前 16 字节）、IV（16~28 字节）、密文（28 字节之后）
2. 用相同的 PBKDF2 参数，以密码 + salt 派生出同一把密钥
3. AES-GCM 解密：用密钥 + IV 解密密文，同时校验认证标签
4. 校验通过则输出明文；校验失败（密码错误或密文被篡改）则抛出异常

## 安全性分析

| 威胁 | 防御机制 |
|---|---|
| 暴力破解密码 | PBKDF2 迭代 10 万次，单次猜测成本高 |
| 彩虹表攻击 | 每次加密使用随机 salt，预计算无效 |
| 密文模式分析 | 随机 IV 保证同明文同密码每次密文不同 |
| 密文篡改 | GCM 认证标签校验，任何改动导致解密失败 |
| 中间人窃听密码 | 纯前端运算，密码不离开浏览器 |

**密钥空间**：AES-256 的密钥空间为 2^256，约 1.16 × 10^77。这远超当前及可预见未来的算力极限，暴力穷举在物理上不可行。

**实际瓶颈在密码强度**：算法本身无法被暴力突破，但如果用户设置的密码过于简单（如 `123456`），攻击者可以通过字典攻击在合理时间内猜到密码。建议使用 12 位以上、包含大小写字母 + 数字 + 符号的密码。
